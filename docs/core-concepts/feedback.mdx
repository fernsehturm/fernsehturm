# Feedback

Feedback is structured critique that drives iteration. It is stored as a first-class artifact type.

## What feedback is

A feedback artifact captures a specific issue with the system—a missing requirement, an inconsistency, a failed test, a policy violation, or a design concern.

Feedback has:

- A **severity** (error, warning, suggestion)
- A **target** (the artifact it criticizes)
- A **body** (the specific issue or recommendation)
- **Links** to related artifacts (e.g., the requirement that is violated)

## Sources of feedback

Feedback can be generated by:

- The **consistency engine** (`fst consistency`)
- The **verification system** (`fst verify`)
- A **human reviewer**
- An **agent** reasoning about the system

## How feedback drives iteration

Feedback does not automatically change anything. It creates a record of an issue. Agents read feedback artifacts and use them to propose revised artifacts that address the issues.

This creates a loop:

```
propose → critique → feedback → revise → propose ...
```

## Why structured feedback?

Unstructured feedback (comments, chat messages) disappears. Structured feedback artifacts persist in the database, are linked to their targets, and can be queried. Agents can systematically work through a backlog of feedback rather than re-discovering the same issues on every iteration.
