# First Controlled Change

The following example shows what working with FST feels like.

The goal is not to learn the internal model. The goal is to see how little you need to do while FST keeps the work controlled.

## Scenario

You want to add passwordless login, but you do not want the agent to quietly rewrite authentication.

Start with a plain request:

```text
/fst Add passwordless login by email magic link.
Keep the existing password login and session behavior working.
Do not add SMS or social login.
```

This is a good FST task because it has real risk:

- auth behavior must not drift
- existing sessions must keep working
- token expiration needs a product decision
- email templates may enter scope
- security checks matter
- the final output should be reviewable

## 1. FST Selects The Accepted Work Context

FST first determines where Exploration may look.

It may find more than one relevant context:

```text
I found:

1. current login/session system
2. old experimental invite-token flow

Use only the current login/session system as the Accepted Work Context?
```

You answer:

```text
Yes. Use the current login/session system.
Do not use the old invite-token flow.
```

This does not approve the implementation. It only selects the starting context.

## 2. FST Asks Only Real Product Questions

The agent can inspect, draft, test, and revise without asking you about every detail.

FST interrupts when your answer changes the product, risk, or allowed path.

Example:

```text
Should a magic link create a new session immediately,
or require confirmation on the same device that requested it?
```

You answer:

```text
Require same-device confirmation.
```

Another example:

```text
Magic links need an expiration time.
Use 15 minutes, 30 minutes, or another value?
```

You answer:

```text
Use 15 minutes.
```

FST records those answers against the exact questions, so the agent cannot later reduce them to "the user approved the login plan."

## 3. FST Blocks Silent Scope Drift

During Build, the agent discovers that the email template must change to include the magic-link URL.

Email templates were not part of the original retained scope, so FST should not let the agent edit them silently.

FST asks:

```text
The implementation needs to modify transactional email templates,
which were not in retained scope.

Expand scope to include transactional email templates?
```

You answer:

```text
Yes. Include only the passwordless-login email template.
```

Now the agent has explicit permission to continue in that area.

## 4. The Agent Works Inside The Recorded Path

The agent can now build the change.

You do not need to drive the internal steps.

FST keeps checking that the work stays inside the recorded path:

- current login/session system is the Accepted Work Context
- password login must keep working
- session behavior must remain unchanged
- SMS and social login are out of scope
- magic-link expiration is 15 minutes
- same-device confirmation is required
- only the relevant email template was added to scope

If the agent tries to change token storage, add SMS, or alter session behavior without evidence, FST should block the work.

## 5. FST Checks Before Materialization

When the proposed work holds together, FST can ask how to project it into concrete output:

```text
The checked result is ready.
Apply it to the workspace or emit a patch?
```

You answer:

```text
Emit a patch.
```

That approval means "write this checked result to this target." It does not mean every future login change is accepted.

## 6. Review The Final Report

At the end, ask:

```text
What changed?
What user decisions were recorded?
What did FST check?
What remains unresolved?
```

A useful final report should look like this:

```text
Added passwordless login by email magic link.

User decisions recorded:
- Use the current login/session system as the Accepted Work Context.
- Magic links require same-device confirmation.
- Magic links expire after 15 minutes.
- Scope expanded to include only the passwordless-login email template.

Checked:
- Existing password login still works.
- Existing session behavior is unchanged.
- SMS and social login were not added.
- Expired magic links are rejected.
- Reused magic links are rejected.
- Email template renders the expected link format.

Materialized:
- Patch emitted.

Remaining:
- No blocking items.
```

The important part is not the exact wording. The important part is that the result can explain what happened without relying on chat memory.

## The Short Version

```text
Ask for the change.
Confirm the Accepted Work Context if asked.
Answer product and risk decisions.
Approve scope expansion only when it is explicit.
Choose how to materialize the checked result.
Review the evidence-backed report.
```
