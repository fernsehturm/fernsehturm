# The Execution Loop

## Core Rule

The worker **never decides the next step.**

Every action comes from the system. The worker's only job is to ask, execute, and report.

---

## The Loop

```text
1. call fst next       → receive the next action
2. execute the action  → run the corresponding command
3. report the result   → submit via fst step submit
4. repeat
```

The system decides what comes next based on the current artifacts, phase, feedback, policies, and runtime results.

---

## Requesting the Next Step

```bash
fst next --json
```

This is the only way the worker progresses. Example response:

```json
{
  "action": "derive_requirements",
  "input": { "request_id": 1 },
  "constraints": ["include failure modes"],
  "continue": true
}
```

The worker must execute this action exactly—no interpretation, no deviation, no batching with other steps.

---

## Executing an Action

The worker translates each action into the corresponding `fst` command:

| Action | Command |
|--------|---------|
| `derive_requirements` | `fst semantic run derive-requirements` |
| `run_consistency` | `fst consistency run` |
| `derive_decisions` | `fst semantic run derive-decisions` |
| `generate_code` | `fst artifact put` |
| `materialize` | `fst materialize` |
| `run` | `fst run` |
| `verify` | `fst verify run` |
| `refine` | `fst semantic run refine` |

These mappings are predefined. The worker does not interpret them freely.

---

## Reporting the Result

```bash
fst step submit --json
```

Even on failure—capture the error and submit it. The system uses the result to decide what happens next.

---

## Full Example

```text
/fst Add login
```

```text
→ fst next → derive requirements  → fst step submit
→ fst next → run consistency      → fst step submit
→ fst next → generate code        → fst step submit
→ fst next → materialize          → fst step submit
→ fst next → verify               → fst step submit
→ fst next → finish
```

Every step is requested, executed, and reported. The worker never skips ahead or combines steps.

---

## Blocking: Ask User

When the system needs human input:

```json
{
  "action": "ask_user",
  "prompt": "Use JWT or sessions?",
  "blocking": true
}
```

The worker stops, asks the user, and resumes only after receiving an answer.

---

## Completion: Finish

When the task is done:

```json
{
  "action": "finish",
  "final_output": "Login system implemented"
}
```

The worker returns the result and stops the loop.
