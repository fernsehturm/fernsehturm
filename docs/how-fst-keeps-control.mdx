# How FST Keeps Control

FST does not keep control by making the agent slow. It keeps control by checking stage exits.

Inside a stage, the agent can inspect, experiment, test, revise, and discard attempts. At the exit, FST checks whether the result is bounded, traceable, falsifiable, and ready for the next kind of evaluation.

## The Control Path

```text
Accepted Work Context
  -> Exploration
  -> Build
  -> Compose
  -> Materialization
```

## Accepted Work Context

The Accepted Work Context answers one question:

```text
Where may Exploration look?
```

It records selected Composition revisions, graph/source revisions, discovery policy, conflict policy, and any required user selection evidence.

It does not grant Build permission.

## Exploration

Exploration answers:

```text
What does this work touch?
```

Its durable output is an `ExplorationNote` with retained scope.

Retained scope is the Build box. It says what may be read, depended on, modified, created, and verified.

If Build later discovers that the box is wrong or too small, FST sends the work back to Exploration instead of letting the agent silently expand scope.

## Build

Build answers:

```text
What Candidate work package was created inside the box?
```

A `Candidate` records the exact artifacts, implementation material, expected touch points, actual touch points, verification definitions, and observations that belong to the work package.

The Build gate checks whether the Candidate stayed inside retained scope and contains enough structure to be composed.

## Compose

Compose answers:

```text
Do these pinned revisions hold together as a possible system world?
```

A `Composition` is not a global accepted state. It is an immutable possible world made from exact revisions.

The Compose gate checks revision consistency, relation validity, contract compatibility, decision uniqueness, policy satisfaction, verification alignment, coverage, freshness, and required user evidence.

If the Composition is coherent, FST has not proven it is the best possible solution. It has proven a precise thing:

```text
This pinned world holds together under the recorded checks and evidence.
```

## Materialization

Materialization projects a coherent Composition into a concrete sink: a workspace, patch, generated directory, migration folder, deployment manifest, or another target.

Materialization is not a fourth stage and not global acceptance. It is a recorded projection of a pinned world.

## The Invariants That Matter To Users

- Work starts from an explicit Accepted Work Context.
- Build can only change retained scope.
- Stored references pin revisions. There is no hidden latest.
- User input that satisfies a gate must be recorded and bound to the exact question.
- Compositions are possible worlds, not global truth.
- Materialized files are projections, not proof by themselves.
- Gate failures create blockers instead of silent state changes.

These rules are why FST can let agents move quickly without letting their output quietly become uncontrolled system behavior.
