# Feedback

Feedback artifacts capture structured critique—issues, gaps, failures, and suggestions that drive iteration.

## Characteristics

- Linked to a specific target artifact
- Have a severity: `error`, `warning`, `suggestion`
- Generated by the system, agents, or humans
- Persist until resolved
- Queryable by agents

## Sources

| Source | How generated |
|--------|--------------|
| Consistency engine | `fst consistency run` |
| Verification system | `fst verify run` |
| Human reviewer | `fst artifact put --type feedback` |
| Agent | `fst artifact put --type feedback` |

## Anatomy of a feedback artifact

```json
{
  "type": "feedback",
  "severity": "error",
  "body": "Requirement REQ-42 has no linked code artifact. It is unimplemented.",
  "links": {
    "targets": ["requirement:<req-id>"]
  }
}
```

## Creating feedback manually

```bash
fst artifact put \
  --type feedback \
  --body "The error handling in tasks.js does not cover the database timeout case" \
  --links code:<code-id>
```

## Resolving feedback

Once an agent or human has addressed the issue, mark the feedback as resolved:

```bash
fst feedback resolve --id <feedback-id>
```

Resolved feedback is retained in history but excluded from active consistency checks.
