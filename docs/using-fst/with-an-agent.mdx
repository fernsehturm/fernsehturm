# Using FST With An Agent

You do not need a special way of thinking to use FST with a coding agent.

Give the agent the task. Let the agent work. Let FST control the stage exits.

## What The Agent Should Do

The agent should:

- record the request
- establish the Accepted Work Context
- explore only the allowed search universe
- promote relevant findings into retained scope
- build Candidate work inside that scope
- run checks and record Observations
- compose pinned revisions
- report blockers instead of hiding them
- ask for user input only when FST requires evidence

## What The Agent Should Not Do

The agent should not:

- treat workspace files as proof of system truth
- use "latest" as a stored reference
- silently expand scope
- treat an agent summary as user approval
- skip Compose by materializing a Candidate as if it were coherent
- use docs as an API contract during controlled work when runtime help is available

## What You Should Ask For

Good user prompt:

```text
/fst Add session expiry after 30 minutes.
Do not introduce JWT refresh tokens.
Use the normal FST control loop and stop only for context choice,
behavior decisions, policy exceptions, scope expansion, or materialization approval.
```

Good end-of-task question:

```text
Show me the Accepted Work Context, retained scope, Candidate, Composition,
checks, user evidence, materialization result, and remaining blockers.
```

## When The Agent Stops

The agent should stop when FST returns a real blocker:

- missing context selection
- stale Exploration
- scope expansion required
- missing user evidence
- decision conflict
- policy conflict
- coverage gap
- judged check required
- materialization approval required

That stop is not a failure. It is where user control belongs.
