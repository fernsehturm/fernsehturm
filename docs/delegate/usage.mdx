# Usage & Best Practices

---

## When to Use the Delegate

**Enable it when you want:**
* to work at a higher level of abstraction
* fewer manual decisions
* automatic critique and gap detection
* faster iteration on features

**Disable or weaken it when you want:**
* full control over every decision
* minimal interruptions
* strict manual review

---

## Common Patterns

### High-level building
Set profile to `assisted_build`. Describe the goal. Let the delegate fill gaps and suggest decisions while you focus on direction.

### Controlled development
Set profile to `review_heavy`. The delegate asks questions but does not answer them—you make every call.

### System testing (recommended early)
Set profile to `internal_full`. Run a large task. Observe what the delegate flags. Repeated critiques usually reveal missing requirements or weak prompts.

### Fine-tuned control
Customize the profile to allow gap-filling but disallow decision acceptance. Useful when you trust the delegate's analysis but not its judgment calls.

---

## Best Practices

**Let the delegate handle routine decisions.** Don't micromanage trivial details—that's what the delegate is for.

**Keep decisions explicit.** Even when the delegate suggests a direction, accept or reject it clearly rather than letting it drift into accepted state.

**Watch recurring feedback.** If the same critique appears across multiple iterations, it points to a missing requirement or an underspecified goal—fix it upstream.

**Use a stronger delegate early.** Then reduce its authority as the system stabilizes. The delegate is most valuable when the shape of the system is still uncertain.

**Trust, but verify.** The delegate helps you think. You still own the final system.

---

## Common Pitfalls

| Pitfall | What happens | Fix |
|---------|-------------|-----|
| Too strong delegate too early | system drifts without clear intent | start with a goal, then enable the delegate |
| Too weak delegate | you do all the product work manually | use at least `assisted_build` |
| Ignoring feedback | same issues surface repeatedly | treat delegate output as required input |
| Mixing roles | delegate suggestions treated as accepted decisions | always explicitly accept or reject |
